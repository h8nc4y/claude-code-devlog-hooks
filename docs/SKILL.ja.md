# 開発日誌の規律(claude-code-devlog-hooks)日本語完全版

これは [SKILL.md](../SKILL.md)(英語・正典)の日本語完全版です。両者は同じ内容を
保ちます。差分を見つけたら英語版を正として issue で報告してください。

コーディングエージェントとして(またはエージェントと共に)働きながら開発日誌を
つけ続けるための規律と、その習慣を機械的に支える Claude Code hook 3本の説明です。
規律そのものはツール非依存で、指示を読めて Markdown に追記できるエージェントなら
何でも従えます。強制層(hooks)だけが Claude Code 専用です。

## なぜ日誌か、なぜ hook か

エージェントセッション中に得た学び — 環境の癖、失敗したアプローチ、決定とその
理由 — はセッションが終わると蒸発します。日次の日誌はそれを検索可能な記録に変え、
蒸留されたトピックノートは繰り返しの学びを「最初に参照する索引」に変えて、同じ轍を
踏むことを防ぎます。

失敗パターンは常に同じです。「最後にまとめて書こう」→ コンテキストが埋まり、
セッションが終わり、何も書かれない。hook 3本はまさにこの失敗パターンを潰すために
あります:

| 層 | hook イベント | 挙動 |
| --- | --- | --- |
| 運用注入 | SessionStart | この規律をコンテキストに注入し、セッション開始時刻をマーカー保存 |
| 催促 | UserPromptSubmit | セッションが一定時間経過 かつ 当日ログが未更新(既定どちらも20分)のときだけ、ブロックせず追記を促す |
| 最低保証 | Stop | セッション開始以降に当日ログが未更新のままターンを終えようとすると、一度だけブロックして記入を求める。以後は邪魔しない |

## 運用ルーチン

1. **着手前**: 日誌の `topics/` フォルダを検索し、タスクに関係する過去の轍を
   確認する。作業中に詰まったときも、力技の前にもう一度検索する。
2. **作業中 — こまめに何度でも書く。** 次のたびに当日ログ
   (devlog ルート配下の `daily/YYYY-MM-DD.md`)へ1項目追記する:
   - 学びを得た(ツールの癖、環境差、落とし穴)
   - 詰まりを解決した(何が効いたか、何が効かなかったか)
   - 方針が決まった(理由も)
   - 切りのいい区切りがついた
   セッションの最後にまとめて書かない。「ログが古い」と促されたり(催促)、
   ターン終了時にブロックされたり(最低保証)したら、それは今のセッションに
   未記録の学びがある合図です。
3. **蒸留する。** 同じ学びを2回書いていると気づいたら、一般化した形を
   `topics/<kebab-slug>.md`(1テーマ1ファイル)へ移し、daily からは
   `[[wikilink]]` でリンクする。daily は時系列の記録、topics は恒久の索引。

## 記入フォーマット

セッション見出しの下に追記します(初回に見出しを作り、以後はその下に足す):

```markdown
## セッション(HH:MM) 〔1行要約〕
- **やったこと**: 実施内容、対象プロジェクト、主な変更
- **学び・詰まり・解決**: 遭遇したエラー、効いた対処、効かなかった対処
- **次回**: 次の最小アクション
- 関連: [[topic-slug]] ／ #tag
```

指針:

- 未来の自分が検索できる粒度で書く。エラー文字列そのもの・フラグ名・バージョン
  番号は、きれいな散文より価値がある。
- 成功だけでなく失敗と行き止まりも書く — 後で一番時間を節約するのはそれ。
- 本当に軽微なセッションは一行で可: 「軽微: <要旨>」。
- secret・token・OAuth 情報・実ユーザー/顧客データは書かない。値が必要に見える
  ときは、値ではなく原因と構造を書く。

## hook への応じ方

- **SessionStart の注入**: セッション中ずっと有効な常備指示として扱う。
- **催促(UserPromptSubmit)**: 未記録の何かがあれば、いま1項目だけ追記する。
  本当に何も無ければ無視してよい(ブロックしない)。
- **ブロック(Stop)**: エントリ(または軽微一行)を追記してから、通常どおり
  ターンを終える。マーカー比較により、ブロックはセッションにつき最大1回しか
  発火しない。抵抗せず、書けばよい。

## 日誌のレイアウト

```text
<devlog ルート>/          # CLAUDE_DEVLOG_DIR で指定
├── daily/YYYY-MM-DD.md   # 1日1ファイル、中にセッション見出し
├── topics/<slug>.md      # 蒸留した恒久ノート、1テーマ1ファイル
└── .devlog-markers/      # hook が書くセッション開始マーカー
```

レイアウトは規約であって hook の要件ではありません: hook が読み書きするのは
`daily/<今日>.md` の mtime とマーカーファイルだけです。Obsidian vault 内の
フォルダを devlog ルートにすると wikilink がそのまま解決されて便利ですが、
素の Markdown フォルダでも同じに動きます。

## Claude Code 以外での規律の運用

他のエージェント CLI(例: Codex)には hooks 機構が無いため、強制層は移植でき
ません。規律は移植できます: 上記ルーチンをエージェントの常備指示(例:
`AGENTS.md`)に貼り、devlog ルートのパスを明記してください。機械的な最低保証は
失われますが、習慣の構造は保てます。

## hook のインストール

hook の登録(Claude Code `settings.json` への3イベント登録+環境変数
`CLAUDE_DEVLOG_DIR` 1つ)はリポジトリの README を、hook がこの作りである理由
(enforce-once マーカー、フェイルオープン、UTF-8 バイト出力)は
[hook-engineering.md](hook-engineering.md) を参照してください。
